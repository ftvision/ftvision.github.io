小的 Pull Request 是高效代码审查的关键。

## 为什么要保持 PR 小

1. **更容易审查** - 审查者可以更专注、更仔细
2. **更快合并** - 小的改动更容易获得批准
3. **风险更低** - 出问题时更容易定位和回滚
4. **反馈更及时** - 不会因为太大而拖延审查

## 如何拆分大的 PR

1. **按功能拆分** - 一个 PR 只做一件事
2. **先重构后功能** - 把重构和新功能分开
3. **先测试后实现** - 测试和实现可以分开提交
4. **分层提交** - 数据层、业务层、展示层分开

## 多小算小？

一般来说：
- 理想：50-200 行改动
- 可接受：200-400 行
- 太大：400+ 行

当然，这取决于改动的性质。纯删除代码或添加测试可以更大一些。
